## 一、工程现状与挂载位置

### 现有结构

```
StartScene
  └→ PrologueScene（飞鸽传书 + 角色创建）
       └→ 第 1 周入职开场剧情节点（数据驱动，发生在固定行动之前）
            └→ FixedActionScene → 自由行动 → ResultScene → …
```

- `PrologueScene.js`：已存在，读取 `DIALOGS_FIXED.prologue` 的三段 `intro / letter / decideToGo`，全部走 `dialog.show()` 串联，最后跳转至下一节点。
- `DEFAULT_STATE`（`GameState.js`）：目前**没有** `playerProfile` 字段，也没有姓名/性别/初始属性偏向的存储结构。
- `ChoicePanel`：已有 `showVNChoice`，支持多选项，可以直接用于角色创建选择。
- `widgets.js`：没有原生输入框，需要用 Phaser 内置的 `this.add.dom()` 或 `scene.rexUI` 实现文字输入，这是**唯一的新增依赖风险点**。

## 三、开场完整流程（交互节点版）

### 阶段 0：画面进入

* 背景：卧室，清晨。
* 画面中使用已绘制素材：无鸽子的卧室（bedroom.png）。
* 无 HUD，无普通按钮，画面渐入后进入旁白。

---

### 阶段 1：旁白铺垫

用 `dialog.show()` 播放第一段旁白：

> 你刚刚毕业没多久。
> 心理学学位，有用，但也不知道在哪里最有用。
>
> 这几个月，你住在家里。每天做饭、看书、给绿植浇水。
> 生活不难过，但有一种轻飘飘的感觉，像站在岸边，不确定要不要踏进去。

旁白结束后，画面出现鸽子飞入窗边的事件，进入"飞鸽传书"。

---

### 阶段 2：飞鸽传书 · 信件事件

画面：切换成bedroom2.png。

旁白：

> 那天早上，一只鸽子从窗户飞了进来。
> 它停了下来，只把一封信留在书桌上。
> 牛皮纸信封，手写地址，字迹工整得像用尺子比着写的。
> 信封上贴着一枚橙色印章，有一颗星星。

点击（在鸽子位置放一个“打开信件”按钮）后展开信件内容。可以用单独样式的信件文本框呈现。

```text
你好。
星星照护所正在寻找一位照护实习生。这里的孩子大多有自闭症谱系特质。工作不轻松，也没有准确的"做对了"。
但如果你愿意先来看看——照护所的门这周一到周五都开着。
如果你愿意回应这封信，请留下你的基本信息和方便称呼。

星星照护所 陈岚
```

信件读完后，用 `ChoicePanel.showVNChoice()` 给玩家一个微小选择：

```text
说明文字：你把信折回原来的样子。

选项 A：「去看看也无妨。」
效果：正常进入后续信息填写。

选项 B：「这封信有点奇怪，但这正是我需要的。」
效果：记录轻量 flag，后续进入照护所的时候解锁特殊台词：
"你注意到了信封上的印章？那是照护所第一批孩子帮我设计的。"——然后在出现这句台词后，所长好感+5.
```

选择后，进入"回应来信 / 填写照护员信息"阶段。

---

### 阶段 3：回应来信 · 创建照护员信息

这部分不是单独的角色创建界面，而是主角按照陈岚信中留下的联系方式，回复自己的基本信息。这样姓名、性别和初始属性选择可以自然接入剧情。

#### 交互节点 A：输入称呼

```text
你的名字是？

[文字输入框]  [确认]
```

* 实现方式：`scene.add.dom()` 嵌入 HTML `<input type="text" maxlength="6">`。
* 确认后读取输入，写入 `gs.player.name`。
* 若为空，默认填入"你"，避免后续称呼出现空白。
* 风险点：`add.dom()` 需要检查当前 Phaser 配置是否支持 DOM。
* 备用方案：如果 DOM 输入框不可用，改为 3 个预设称呼选项，完全用 `ChoicePanel` 实现。

---

#### 交互节点 B：选择性别 / 外观方案

用 `ChoicePanel.showVNChoice()` 实现。

```text
说明文字：你的性别是？(仅影响后续画面和立绘呈现)

选项 A：女性
选项 B：男性
```

* 写入 `gs.player.gender`。
* 当前阶段只存值，不加载立绘资源。
* 后续如果加入主角立绘，可根据该字段调用不同外观方案。

---

#### 交互节点 C：分配初始属性

初始共有 50 点，可分配到三个属性：

* 专业理解
* 沟通能力
* 体能经验

界面形式：

```text
说明文字：你想怎样整理自己的起点？

专业理解    [-]  20  [+]
沟通能力    [-]  20  [+]
体能经验    [-]  10  [+]

剩余点数：0
[确认]
```

规则：

* 总点数固定为 50。
* 每次点击加号或减号调整 10 点。
* 单项不得低于 0。
* 三项总和必须等于 50 才能确认。
* 确认后写入对应属性。
* 这不是复杂养成系统，只是序章初始倾向，用于让玩家感觉"这是我的照护员"。

---

### 阶段 4：决定出发

填写完成后，播放最后一段旁白：

## 四、数据存储方案

轻量追加 `DEFAULT_STATE`，不改时间系统和核心逻辑：

```js
player: {
  name: '',
  gender: 'fenmale', // 'female' | 'male' | 'neutral'
  attrs: {
    professional: 0,
    communication: 0,
    physical: 0,
  },
  prologueLetterReaction: '',
}
```

说明：

* `name` 来自玩家填写的称呼。
* `gender` 来自性别 / 外观选择。
* `attrs` 记录玩家分配的 50 点初始属性。
* `prologueLetterReaction` 记录玩家对飞鸽传书的反应，用于后续第 1 周所长闲聊的特殊台词。
* 不改 `DayManager`。
* 不改七周推进。
* 不改固定行动、小游戏和结算核心逻辑。

## 五、风险点与备用方案

| 风险                       | 等级 | 备用方案                            |
| ------------------------ | -- | ------------------------------- |
| `scene.add.dom()` 输入框不可用 | 中  | 改为预设称呼选项，使用 `ChoicePanel`       |
| 属性分配 UI 工程量偏大            | 中  | 先用三个预设方案代替手动加减点                 |
| 信件特殊样式不兼容现有 DialogBox    | 低  | 复用 `dialog.show()` 多页展示         |
| 后续文本没有接入玩家称呼替换           | 低  | 先保存字段，后续统一处理 `{player.name}` 替换 |

## 六、序章结束后的衔接

序章最后一段旁白点击完毕后，画面淡出，序章结束。

**跳转目标：第 1 周入职开场剧情节点**（不是直接进入 `OfficeScene` 主界面操作层）。

### 正确的完整流程结构

```
StartScene
  └→ PrologueScene（飞鸽传书 + 角色创建）
       └→ 第 1 周入职开场剧情节点
            └→ FixedActionScene（固定行动阶段）
                 └→ 自由行动（OfficeScene + MapScene）
                      └→ ResultScene
                           └→ 下一天 …
```

### 第 1 周入职开场剧情节点的职责

该节点发生在固定行动阶段之前，用于完成：

1. 主角步入照护所、环境初印象
2. 陈岚接待、第一句对话（可用 `gs.player.name` 个性化称呼）
3. "先观察"新手引导和第一周目标说明
4. 若 `prologueLetterReaction === 'curious'`，所长多说一句关于橙色印章的彩蛋台词

### 挂载建议（数据驱动，不硬编码入 OfficeScene.js）

`dialogs_fixed.js` 中已经存在以下键，直接用于承载该节点内容，不需要新建文件或新建 Scene：

| 键名 | 用途 |
|---|---|
| `DIALOGS_FIXED.day1.office.arrival` | 主角到达照护所的第一段旁白/环境描述 |
| `DIALOGS_FIXED.day1.office.directorGreeting` | 陈岚的接待台词（可内联 `{player.name}`） |
| `DIALOGS_FIXED.day1.office.todayFlowIntro` | 新手引导：今天的流程说明、"先观察"目标 |

`OfficeScene.js` 在 `create()` 时已经读取这些键并顺序播放。**只需在 `dialogs_fixed.js` 中填充文本内容，不在 `OfficeScene.js` 中硬编码任何剧情判断逻辑。**

如果橙色印章彩蛋台词需要条件分支，建议：

```js
// 仅在 dialogs_fixed.js 数据层或 OfficeScene 对话播放器处做一次 gs 读取判断：
if (gs.player.prologueLetterReaction === 'curious') {
  // 追加一条 directorGreeting 台词
}
```

这一条件判断只有一处、只在第一天触发，不涉及新系统。

## 七、软衔接说明

序章存储的玩家数据影响第 1 周体验，但不改变任何核心逻辑：

* `gs.player.name`：所长第一句话用称呼打招呼（空则保持"你"）。
* `gs.player.prologueLetterReaction`：`'curious'` 触发印章彩蛋台词，其余不影响任何分支。
* 初始属性分配：只影响开局数值显示，不影响主线走向、固定行动或结算逻辑。
